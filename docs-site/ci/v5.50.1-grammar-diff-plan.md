## 改文件清单(精确)

### 1. `server/lib/ruleforge-core/src/main/antlr4/com/ruleforge/drl/DrlParser.g4`

#### 1.1 `lhsFrom` 扩 4 alt(A1 联动,行 164-167)

**改前**:
```antlr
lhsFrom
    : lhsAtomic DRL_FROM lhsAtomic
    | lhsAtomic DRL_FROM expr
    ;
```

**改后**:
```antlr
lhsFrom
    : drlPattern DRL_FROM drlPattern
    | drlPattern DRL_FROM expr
    | lhsAtomic DRL_FROM lhsAtomic
    | lhsAtomic DRL_FROM expr
    ;
```

**理由**: 当前 `lhsAtomic` 不收 `drlPattern`,所以 `Applicant(...) from $stream`
永远优先走 `drlPattern` 单 alt 收尾,到不了 `lhsFrom`。先放 `drlPattern` alt 即可解。

#### 1.2 `lhsCollect` 改 `lhsAtomic` → `drlPattern`(A2,行 169-172)

**改前**:
```antlr
lhsCollect
    : lhsAtomic DRL_FROM DRL_COLLECT LPAREN lhsPattern RPAREN
    | DRL_COLLECT LPAREN lhsPattern RPAREN
    ;
```

**改后**:
```antlr
lhsCollect
    : drlPattern DRL_FROM DRL_COLLECT LPAREN lhsPattern RPAREN
    | DRL_COLLECT LPAREN lhsPattern RPAREN
    ;
```

**理由**: 同 A1,`ArrayList()` 是 drlPattern 不是 lhsAtomic。

#### 1.3 `methodChain` `+` → `*` + 单 `methodCall` alt(B4,行 258-260)

**改前**:
```antlr
methodChain
    : methodChainHead (DOT methodCall)+
    ;
```

**改后**:
```antlr
methodChain
    : methodChainHead (DOT methodCall)*
    | methodCall
    ;
```

**理由**: 当前 `+` 强制 1+ call,某些 case(单 methodCall / 没有 methodChainHead)走不通。
**注意**: 此改动也可能破坏 `rhsStatementsGrammarLockIn`(绿 test)— 测试时验证它仍绿。

#### 1.4 `accumulateInit` 重写 + `initBody` 3 alt(B1,行 186-189)

**改前**:
```antlr
accumulateInit
    : DRL_INIT LPAREN expr RPAREN
    | DRL_INIT LPAREN statementBlock RPAREN
    ;
```

**改后**:
```antlr
accumulateInit
    : DRL_INIT LPAREN initBody RPAREN
    ;

initBody
    : expr (COMMA expr)*
    | IDENTIFIER ASSIGN expr (COMMA expr)*
    | statement (SEMI statement)*
    ;
```

**理由**: `init(count = 0)` 实际是 `IDENTIFIER ASSIGN expr`,但当前 grammar 走 `expr` 槽位,
`count = 0` 走 statement 槽位的 `assignStatement: expr ASSIGN expr`,产生 statementBlock 边界冲突。

#### 1.5 `constraint` 加 `IDENTIFIER operator LPAREN exprList RPAREN` alt(A3 联动,行 212-216)

当前 constraint 已有这个 alt,不需要改。**但 stringMethods 的真问题在 lhsFrom 边界,
改完 1.1 后 stringMethods 自动过。** 如果改完还红,需再调整:
- 给 `lhsAnd` 显式加 `stringMethod LBRACK` 终结符 alt
- 或给 `constraint` 加 `arrayMethod` 形态(在 RBRACK 后)

**当前 1.1 改完跑一次 — 仍红再加此步**。

### 2. `server/lib/ruleforge-core/src/main/java/com/ruleforge/ir/drl/DrlAstVisitor.java`

#### 2.1 `visitLhsFrom(ctx)` 新增(已存在 lhsFrom walk,加显式校验)

```java
@Override
public Void visitLhsFrom(DrlParser.LhsFromContext ctx) {
    // V5.50.1: lhsFrom 现在 4 alt,首 alt 走 drlPattern,需要校验 type 已 declared
    if (ctx.drlPattern() != null) {
        visit(ctx.drlPattern());  // 触发 visitDrlPattern 的 type 校验
    }
    return visitChildren(ctx);
}
```

**理由**: `visitLhsUnary` 已有 `if (ctx.lhsFrom() != null) { visit(ctx.lhsFrom()); }`,
但 visitLhsFrom 当前没 override,base visitor 不深入 child。**必须显式 override 才能触发
新加的 `drlPattern` 校验**。

#### 2.2 `visitLhsCollect(ctx)` / `visitLhsAccumulate(ctx)` 类似新增

```java
@Override
public Void visitLhsCollect(DrlParser.LhsCollectContext ctx) {
    if (ctx.drlPattern() != null) {
        visit(ctx.drlPattern());
    }
    return visitChildren(ctx);
}

@Override
public Void visitLhsAccumulate(DrlParser.LhsAccumulateContext ctx) {
    if (ctx.lhsAtomic() != null) {
        visit(ctx.lhsAtomic());
    }
    if (ctx.drlPattern() != null) {
        visit(ctx.drlPattern());
    }
    // accumulate 内部 initBody / statementBlock 暂不展开
    return visitChildren(ctx);
}
```

#### 2.3 `visitRhsConsequence(ctx)` / `visitMethodChain(ctx)` 新增

```java
@Override
public Void visitRhsConsequence(DrlParser.RhsConsequenceContext ctx) {
    // V5.50.1: rhs 多 statement 解析,每条都走
    for (DrlParser.StatementContext s : ctx.statement()) {
        visit(s);
    }
    return null;
}

@Override
public Void visitMethodChain(DrlParser.MethodChainContext ctx) {
    if (ctx.methodChainHead() != null) {
        visit(ctx.methodChainHead());
    }
    for (DrlParser.MethodCallContext mc : ctx.methodCall()) {
        visit(mc);
    }
    if (ctx.methodCall() != null && ctx.methodChainHead() == null) {
        visit(ctx.methodCall());
    }
    return null;
}
```

public class FromLeftPart extends AbstractLeftPart {
    private String property;

#### 3.3 `extractLhs` 改 — PENDING_LHS caller 切到 Rule.lhs.criterion 链

**改前**(L269-281):
```java
private static Lhs extractLhs(ParsedDrlRule p, DatatypeResolver resolver) {
    Lhs lhs = new Lhs();
    if (p.getLhsParseTree() == null) {
        return lhs;
    }
    List<PropertyCriteria> pcs = new LhsPropertyVisitor(resolver).extract(p.getLhsParseTree());
    if (!pcs.isEmpty()) {
        PENDING_LHS.put(p.getName(), pcs);  // 静态 map 写
    }
    return lhs;
}
```

**改后**:
```java
private static Lhs extractLhs(ParsedDrlRule p, DatatypeResolver resolver) {
    Lhs lhs = new Lhs();
    if (p.getLhsParseTree() == null) {
        return lhs;
    }
    LhsPropertyVisitor v = new LhsPropertyVisitor(resolver);
    v.extract(p.getLhsParseTree());
    // V5.50.1 migration:from / collect / accumulate 产 FromLeftPart / CollectLeftPart /
    // CommonFunctionLeftPart 直接挂到 Lhs.criterion 链,不再 PENDING_LHS
    if (!v.getCriteria().isEmpty()) {
        lhs.setCriterion(v.getCriteriaJunction());  // 假设有 Criterion 接口
    }
    return lhs;
}
```

**`PENDING_LHS` 字段 + `getPendingLhsCriteria` 方法标 @Deprecated, V5.51 删**。
**V5.50.1 不删**,因为 caller(KnowledgeBuilder / RulesRebuilder)还在用 — `PendingLhsMigration`
测试已经会失败,等 caller 全部切走后,下个 sprint 删字段。
